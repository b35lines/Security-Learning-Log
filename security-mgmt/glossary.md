## 1. ネットワーク防御の三銃士
「どこで」「何を」守るかの違いが重要。

| 用語 | 守備範囲 | 主な役割 | イメージ |
| :--- | :--- | :--- | :--- |
| **FW (ファイアウォール)** | ネットワーク境界 | IPアドレスとポートで通していい通信か判断する。 | 玄関の門番 🚪 |
| **IPS (侵入防止システム)** | ネットワーク全体 | 通信の中身をリアルタイム監視し、攻撃を遮断する。 | 館内パトロール 👮 |
| **WAF** | Webアプリ | SQLインジェクション等、Web特有の攻撃を深くチェック。 | 受付の専用検査官 🌐 |

## 2. 攻撃と対策のキーワード
最新トレンドや「おとり」の技術。

- **サプライチェーン攻撃** 📦
    - 信頼している取引先やソフトウェアの更新ルートを悪用して侵入する手法。
- **ハニーポット** 🍯
    - あえて脆弱性を見せて攻撃者を誘い込み、手法を観察するための「おとり」システム。
- **レジリエンス** 🌱
    - 攻撃を100%防ぐのではなく、被害を受けても「素早く回復する」という考え方。

## 3. 組織・管理のルール
「人」による不正やミスを防ぐ仕組み。

- **職務分掌 (SoD)** 👥
    - 「申請」と「承認」のように役割を分け、一人で不正ができないようにすること。
- **最小権限の原則** 🔑
    - 業務に必要な「最低限」の権限だけを与えること（不要な権限は持たせない）。
- **プロキシサーバー** 代理
    - 社内から外へのWebアクセスを中継し、ログ記録や有害サイト遮断を行う。

## 🔍 ログと構成の可視化 (SIEM & SBOM)
「何が起きているか」「何でできているか」を把握するための技術。

| 用語 | 読み方 | 役割 | 導入するメリット |
| :--- | :--- | :--- | :--- |
| **SIEM** | シーム | 各種機器のログを集約・相関分析する。 | 単独のログでは気づけない「攻撃の流れ」を検知できる。 |
| **SBOM** | エスボム | ソフトウェアに含まれる部品（ライブラリ等）の一覧表。 | 脆弱性が見つかった際、自社ソフトに影響があるか即座に判断できる。 |

### 🛠️ SIEM (Security Information and Event Management) 🚨
- **仕組み**: FW、サーバー、PCなどのバラバラなログを1箇所に集めて、リアルタイムで分析する。
- **なぜ必要か**: 
    - 攻撃者は巧妙に侵入するため、一つの機器のログだけでは異常に見えないことが多い。
    - 「深夜に普段使わない端末が(PCログ)」「大量のデータを外部へ送信した(FWログ)」といった**点と点を結んで**、攻撃をあぶり出す。
- **イメージ**: 建物中の防犯カメラとセンサーを一括管理する「中央監視室」。

### 🛠️ SBOM (Software Bill of Materials) 📋
- **仕組み**: ソフトウェアの「成分表示表」や「お品書き」。
- **なぜ必要か**: 
    - 現代のソフトは多くの「オープンソース（外部部品）」を組み合わせて作られている。
    - 外部部品に欠陥（脆弱性）が見つかったとき、SBOMがないと「自社のどのソフトが危険か」を調べるだけで数週間かかってしまう。
- **イメージ**: 食品の「原材料ラベル」。アレルギー物質（脆弱性）が含まれていないか一目でわかる。

### 📦 サプライチェーン攻撃 (Supply Chain Attack)
ターゲットを直接狙わず、その「供給網（サプライチェーン）」の弱点を突く攻撃。

#### 主な攻撃パターン
1. **ビジネスパートナー経由** 🤝
   - ターゲット企業の「子会社」や「委託先」など、セキュリティが比較的甘い組織をまず踏み台にする。
2. **ソフトウェア・サプライチェーン経由** 👨‍💻
   - 開発ツールや、OS・アプリの「アップデート用サーバー」にウイルスを仕込む。利用者が正規の更新を行うだけで感染する。
   - 関連用語: **SBOM** (部品表で管理することで、汚染された部品を特定する)
3. **ITサービス経由** ☁️
   - 利用しているクラウドサービスや保守運用会社を攻撃し、そこから芋づる式に利用企業の情報を盗む。

#### 試験対策のポイント
- **「信頼の悪用」**: 正規のルートや更新プログラムを悪用するため、検知が難しい。
- **「ゼロトラスト」の必要性**: 「身内（取引先や正規ソフト）だから安全」という考えを捨て、常に検証する。

 ### 🧪 DNSキャッシュポイゾニング
キャッシュサーバーに「偽の住所（IPアドレス）」を覚えさせ、ユーザーを偽サイトへ誘導する攻撃。

#### 1. DNSの基本用語
- **キャッシュサーバー** (案内窓口): PCからの依頼で代わりに住所を調べて、結果を一時保存する。
- **権威DNSサーバー** (名簿管理者): ドメインの正確な住所を知っている正式なサーバー。
- **再帰的問い合わせ**: PC → キャッシュサーバー。「住所を調べてきて！」という丸投げ。
- **非再帰的問い合わせ**: キャッシュサーバー → 権威サーバー。「知ってたら教えて」という確認。

#### 2. 攻撃のステップ
1. 犯人がキャッシュサーバーに「偽の回答」を大量に送りつける。
2. キャッシュサーバーが、本物からの回答よりも先に偽物を受け取り、それを「正解」として保存（キャッシュ）してしまう。
3. 以降、そのサーバーを使うユーザーは全員、偽サイトに飛ばされる。

#### 3. 対策
- **DNSSEC**: 応答にデジタル署名を付け、「これは本物の名簿管理者が書いた回答です」と証明する仕組み。
- **ソースポートランダム化**: 問い合わせに使うポート番号をバラバラにして、犯人が偽物を送り込む難易度を上げる。
- **外部のDNSクライアントからの再帰的問い合わせを受け付けない設定にする**

### 📊 CVSS v3 (共通脆弱性評価システム)
脆弱性の深刻度を 0.0 〜 10.0 で表す世界標準の物差し。

#### 3つの評価基準
1. **基本評価基準 (Base)** 🏗️
   - **内容**: 脆弱性そのものの性質（攻撃の難易度や影響度）。
   - **特徴**: 時間や環境で**変化しない**。メーカーが公表する「公式スコア」。
2. **現状評価基準 (Temporal)** ⏳
   - **内容**: 攻撃コードの流通状況や、修正プログラムの有無。
   - **特徴**: 時間とともに**変化する**。パッチが出ればスコアは下がる。
3. **環境評価基準 (Environmental)** 🏢
   - **内容**: 組織内でのシステムの重要度や、セキュリティ対策の適用状況。
   - **特徴**: 組織ごとに**変化する**。最も「自分たちへの影響」を反映する。

> **💡 実務での流れ**:
> 「基本」で深刻なものを見つける → 「現状」で緊急性を判断する → 「環境」で自社での対応優先度を決める。
####  CVSSスコアの読み解き方（例）
- **Base 10.0 / Temporal 10.0 / Environmental 10.0**
  → 非常に危険。かつ対策（パッチ）もなく、自社の重要サーバー。**即座に対応が必要。**
- **Base 10.0 / Temporal 8.0 / Environmental 0.0**
  → 脆弱性は最悪だが、パッチは存在し、かつ自社では未使用の隔離端末。**後回しでOK。**

### 🕵️ IDS（侵入検知システム）の分類
異常を検知して管理者に通知するシステム。設置場所で2種類に分かれる。

| 種類 | 正式名称 | 監視場所 | 特徴・メリット |
| :--- | :--- | :--- | :--- |
| **NIDS** | Network IDS | ネットワークの境界 | 1台でネットワーク全体を監視できる。 |
| **HIDS** | Host IDS | サーバーやPC内部 | **暗号化された通信**経由の攻撃も、OSレベルの挙動で検知できる。 |

### 🛡️ IDSとIPSの違い
不正な通信に対する「アクション」が異なる。

| 特徴 | IDS (侵入検知) | IPS (侵入防止) |
| :--- | :--- | :--- |
| **主な動作** | 通知（アラート）のみ | **遮断（ドロップ）**と通知 |
| **設置方法** | 経路の横（ミラー/コピー） | 経路の途中（**インライン**） |
| **故障時の影響** | 通信は継続する | 通信が止まる可能性がある |
| **誤検知の影響** | ログが汚れるだけ | **正規の通信が遮断される** |

> **⚠️ 試験のポイント**:
> - 「暗号化通信」には弱い（NIDS/ネットワーク型の場合）。
> - 「シグネチャ型（既知の攻撃）」と「アノマリ型（未知の異常）」の検知方式がある。

### 🎣 CSRF (Cross-Site Request Forgeries)
ログイン中のユーザーのブラウザを利用して、本人の意図しない操作（送金、パスワード変更等）を強制させる攻撃。

#### 🛡️ 主要な対策：ワンタイムトークン
- **仕組み**: サーバーが生成した「推測困難な使い捨ての合言葉（トークン）」を、リクエストに含めて送信させる。
- **チェック**: サーバー側で「自分が発行したトークンと一致するか」を確認し、一致しない場合は処理を拒否する。
- **ポイント**: 
    - URLパラメータではなく、**メッセージボディ（POST）** や **HTTPヘッダー** で送信する。
    - 罠サイトからは「同一起源ポリシー」により、トークンの値を読み取ることができないため攻撃を防げる。

# 🛡️ ネットワーク攻撃（Smurf, SYN Flood, IPスプーフィング）まとめ

## 1. Smurf攻撃 (Smurf Attack)
ICMPの応答の仕組みを悪用し、ターゲットの通信回線をパンクさせる攻撃です。

* **仕組み** ⚙️
    1. 攻撃者が、送信元IPアドレスを「ターゲットのIP」に偽装します（IPスプーフィング）。
    2. 偽装したパケットを、ネットワーク内の全デバイスに届く「ブロードキャストアドレス」宛てに送信します。
    3. 信号を受け取った大量のデバイスが一斉にターゲットへ応答（Echo Reply）を返し、ターゲットの帯域を占有します。
* **特徴** 📢
    - 攻撃パケットがネットワーク内で「増幅」されるため、少量の通信で大きな被害を与えます。
* **対策** 🛡️
    - 外から入ってくるブロードキャスト宛てのパケットをルーターで遮断する。

## 2. SYN Flood攻撃
TCPの接続開始手順（3ウェイ・ハンドシェイク）の隙を突く攻撃です。

* **仕組み** 🤝
    1. 攻撃者が大量の接続要求（SYNパケット）をサーバーに送ります。
    2. サーバーは応答（SYN/ACK）を返し、相手からの最終確認（ACK）を待つ「ハーフオープン」状態になります。
    3. 攻撃者はACKを返さず放置するため、サーバーの接続待機枠（メモリリソース）がいっぱいになり、正規のユーザーが接続できなくなります。
* **対策** 🛡️
    - **SYN Cookies**: 接続が完全に確立するまでサーバー側に情報を保持しない技術。
    - タイムアウト（待機時間）の設定を短くする。

## 3. IPスプーフィング (IP Spoofing)
送信元IPアドレスを偽装する技術で、多くの攻撃の「準備」として使われます。

* **仕組み** ✍️
    - IPパケットの「ヘッダー」を書き換え、あたかも信頼されたIPや、別のターゲットのIPから送られたように見せかけます。
* **用途** 🎯
    - 攻撃者の正体を隠す。
    - Smurf攻撃のように、ターゲットへ「返信」を集中させるためのトリガーにする。
* **対策** 🛡️
    - **イングレスフィルタリング**: 内部ネットワークから出るパケットが、自組織のものでないIPを名乗っている場合に遮断する。

# FIDO (Fast IDentity Online) 認証の仕組み

FIDOは、パスワードを使わずに「生体情報」や「デバイスの所有」を利用してログインする、安全な認証規格です。

## 🔑 基本的な仕組み：公開鍵暗号方式の活用

FIDOでは、ネットワーク上に秘密の情報を流さないために「公開鍵暗号方式」を利用しています。

### 1. 登録時
ユーザーのデバイス（スマートフォンやセキュリティキー）内で、対になる2つの鍵を生成します。
- **秘密鍵 🗝️**: デバイス内の安全な領域（セキュアエレメントなど）に保管され、外には絶対に出しません。
- **公開鍵 🔓**: ログイン先のサーバーへ送り、登録します。

### 2. 認証時
1. **チャレンジの受信**: サーバーからランダムな使い捨てデータ（問題）が送られてきます。
2. **本人確認**: ユーザーが指紋や顔認証を行い、デバイスが「本人である」と確認します。
3. **署名作成**: デバイス内の秘密鍵を使って、サーバーからの「問題」に対する「回答（デジタル署名）」を作成し、サーバーへ送り返します。
4. **検証**: サーバーは登録済みの「公開鍵」を使って、署名が正しいかを確認します。

## 🛡️ 安全性のポイント

- **プライバシー保護**: 指紋や顔などの生体データそのものはデバイス内でのみ照合され、サーバーには送られません。
- **フィッシング耐性**: デバイスがアクセス先のドメイン（URL）を確認するため、偽サイトに対しては署名を行わない仕組みになっています。

# マルウェア検知手法のまとめ

マルウェア（悪意のあるソフトウェア）を検知するための主な3つのアプローチについて整理します。

## 1. 静的検知（シグネチャ法 / チェックサム） 🔍
ファイルの「中身」を実行せずに解析する手法です。

- **仕組み**: 既知のマルウェアのハッシュ値（チェックサム）や、特定の特徴的なコードパターン（シグネチャ）のデータベースと照合します。
- **メリット**: 処理が高速で、既知の脅威に対して非常に正確。
- **デメリット**: 未知のマルウェア（ゼロデイ攻撃）や、コードを少しだけ書き換えた亜種には対応できません。

## 2. 動的検知（ビヘイビア法 / 振る舞い検知） ⚙️
ファイルを実際に「動かしてみて」その挙動を観察する手法です。

- **仕組み**: サンドボックスと呼ばれる隔離された安全な仮想環境でプログラムを実行し、「勝手に外部へ通信していないか」「重要なシステムファイルを書き換えていないか」といった振る舞いを監視します。
- **メリット**: コードが難読化されていても、実際の動作から悪意を判断できる。
- **デメリット**: 実行して観察するため時間がかかる。また、仮想環境であることを検知して動作を止める巧妙なマルウェアも存在します。

## 3. 推論検知（ヒューリスティック法） 🧠
「怪しいプログラムがやりそうなこと」を確率的に判断する手法です。

- **仕組み**: 特定のルールやAI（機械学習）を用いて、「このプログラムは自己増殖するコードを持っている」「不自然な難読化がされている」といった特徴から、マルウェアである可能性をスコアリングします。
- **メリット**: 未知のマルウェアや亜種を、実行する前にある程度予測して防ぐことができる。
- **デメリット**: 正解ではないのに「怪しい」と判定してしまう「誤検知（フォールスポジティブ）」が発生する可能性があります。

# ファジング (Fuzzing / Fuzz Testing)

ファジングは、ソフトウェアの脆弱性やバグを発見するために、予測不能なデータ（ファズ）を大量に入力して挙動を監視する「動的解析」の手法です。

## 📖 語源と定義
- **英語**: Fuzzing / Fuzz Testing
- **Fuzzの由来**: もともとは「綿毛」や「ぼやけた」という意味ですが、IT分野では「意味を持たない、あるいは予期しないノイズのようなデータ」を指します。

## ⚙️ 仕組みとプロセス
1. **データ生成**: テスト対象が想定していない、無効なデータやランダムなデータ（ファズ）を作成します。
2. **データ入力**: 対象のプログラムに自動で繰り返し入力します。
3. **監視**: クラッシュ、フリーズ、メモリの異常な消費、エラーの発生などをチェックします。
4. **解析**: 異常が発生した場合、どの入力データが原因で、どのような脆弱性（バッファオーバーフローなど）があるかを特定します。

## 🛡️ メリットと重要性
- **未知の脆弱性の発見**: 開発者が想定していない「盲点」を突くため、ゼロデイ脆弱性の発見に非常に有効です。
- **自動化**: ツール（ファザー）を使うことで、人間が手動で行うには膨大な時間がかかるテストを24時間実行できます。
- **コスト対効果**: 比較的単純な仕組みで、深刻なセキュリティホールを効率よくあぶり出すことができます。

## 16進数の桁数とビット（スイッチ）の関係
- **16進数とビット**: 
  - 16進数1文字（16種類）を表すには、最低 **4ビット** ($2^4=16$) が必要。
  - 16進数2文字（例: `ff`）は **8ビット（1バイト）** で表現される。
16進数の桁が増えるとき、必要なビット数は「足し算」で増えていきます。これをスイッチの数に例えると理解しやすくなります。

### 💡 桁が増えるときの考え方
16進数1文字（0〜f）を表現する「箱」をイメージしてみましょう。

- **1文字目（左の桁）**: 
  - 0〜f の16種類を区別するために **4つのスイッチ（4ビット）** が必要です。
- **2文字目（右の桁）**: 
  - 同様に 0〜f の16種類を区別するために、さらに **4つのスイッチ（4ビット）** が必要です。

### 🔢 合計の計算
これらを横に並べると、全体で必要なスイッチの数は以下のようになります。
- **計算式**: $4 \text{ビット} + 4 \text{ビット} = 8 \text{ビット}$

このように、桁が増えるごとに「4ビットずつ」スイッチを足していくことで、より大きな数（より多くの組み合わせ）を表現できるようになります。

# ステガノグラフィ (Steganography)

データの中に別のデータを「隠蔽」する技術。情報の存在自体を気づかせないことを目的とする。

## 🔑 暗号（Cryptography）との違い
- **暗号**: 内容を解読不能にする（「何か書いてあるが読めない」状態）。
- **ステガノグラフィ**: 存在を隠す（「何も書いていないように見える」状態）。

## 🖼️ 代表的な手法：LSB法 (Least Significant Bit)
- デジタルデータの最下位ビットを秘密の情報で書き換える。
- 人間の目や耳には変化が知覚できない程度の微細な改ざんを利用する。

# EDR (Endpoint Detection and Response)

エンドポイント（PCやサーバー）における挙動を監視し、サイバー攻撃を検知・対応するソリューション。

## 🔑 EPP（従来型）との違い
- **EPP (Endpoint Protection Platform)**: 侵入を「未然に防ぐ」ことが目的。
- **EDR**: 侵入された後の「検知・対応・復旧」が目的。

## ⚙️ 主な機能
1. **監視 (Monitoring)**: プロセス、ファイル、ネットワーク通信のログを常時記録。
2. **検知 (Detection)**: シグネチャに頼らず、振る舞い（アノマリ）から不審な動きを特定。
3. **封じ込め (Containment)**: 感染した端末をネットワークから隔離し、被害拡大を防止。
4. **調査 (Investigation)**: 攻撃がどこから入り、何をしたのかのルート（根本原因）を解析。

- **データ漏洩**: 持ち出し禁止の情報を画像に埋め込んで外部に送信する。
